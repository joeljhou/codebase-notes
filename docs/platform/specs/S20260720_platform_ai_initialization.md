# 使用 AI 初始化 Codebase Notes

> 状态：已完成
> 最后更新：2026-07-20
> 适用范围：Codebase Notes v1、VS Code、TRAE、JetBrains IDE
> 关联文档：[Codebase Notes v1 协议规范](S20260717_platform_codebase_notes_protocol.md)

把本文完整交给能够读取项目文件的 AI，即可让它创建或更新仓库根目录的 `.codebase-notes.json`。

## 1. 你的任务

分析当前代码仓库，在仓库根目录创建或更新 `.codebase-notes.json`。

这份文件是项目树旁边的“代码地图”。备注应该像路牌：短、准、扫一眼就懂。不要把它写成目录说明书，也不要给每个文件添加备注。

完成任务时必须遵守本文的调查、选择、文字、颜色、格式、合并和校验规则。

## 2. 先调查，再生成

优先阅读以下高信号内容：

1. `AGENTS.md`、`CLAUDE.md` 等仓库指令。
2. 根目录和主要模块的 README。
3. 依赖清单、构建配置、工作区配置和启动脚本。
4. 前后端启动入口、路由、模块装配和依赖注入位置。
5. 核心领域模型与主要业务链路。
6. API、Schema、Proto、数据库模型和其他权威契约。
7. 插件、Provider、Adapter、Handler、Hook 和注册表。
8. 测试、构建、发布、部署和代码生成入口。

必须从文件内容、导入调用、构建配置或仓库文档中验证结论。禁止只根据文件名猜测。

## 3. 只选择真正有导航价值的路径

优先备注：

- 项目或应用主入口。
- 核心业务链路和核心领域模型。
- 重要模块、API 边界和工程入口。
- 唯一事实来源、权威契约和稳定基准。
- 新增能力时需要使用的扩展点。
- 容易忽略的生成规则、兼容关系和同步要求。
- 测试、构建、发布和部署入口。

不要备注：

- 一眼就能从名称看出用途的普通文件。
- `node_modules`、构建缓存、IDE 缓存和临时文件。
- 普通锁文件、生成产物中的每个子文件。
- 只有“这是源码目录”“这是测试目录”这类重复路径名称的信息。
- 无法从仓库证实的架构猜测。

数量建议：

- 小型项目：8～15 条。
- 中型项目：15～30 条。
- 大型项目或 monorepo：30～50 条。
- 宁可少，不要为了覆盖目录树凑数量。

## 4. 备注文字规范

- 默认使用简体中文，代码标识符保持原样。
- 使用短语，不写完整说明文。
- 推荐 4～16 个汉字，硬上限 24 个 Unicode 字符。
- 每条只表达一个最重要的信息。
- 不使用句号、星号、emoji 或人工强调符号。
- 不以“该文件”“该目录”“这个文件”“主要负责”开头。
- 不复述文件名已经表达的信息。
- 优先写职责、阅读顺序、权威性、扩展方式或隐藏关系。
- 如果有价值的信息无法压缩成短备注，宁可不添加。

推荐：

```text
前端主入口
订单核心链路
API 权威契约
新增 Handler 注册
由 OpenAPI 自动生成
```

不推荐：

```text
该文件主要负责整个前端应用的初始化和各种依赖的装配工作。
★ 非常重要的核心文件
这个目录里面包含了所有用于处理订单业务逻辑的相关代码。
```

## 5. 颜色心智

颜色用于构建代码地图，不表示错误、警告或运行状态。

必须始终使用同一套心智：

> 红色看主干，黄色看细节，蓝色看重点，绿色找基准，紫色找扩展。

| `style` | 名称与颜色 | 用户心智 | 典型对象 |
| --- | --- | --- | --- |
| 省略或 `default` | 普通，灰色 | 知道它做什么即可 | 普通职责说明 |
| `important` | 重要，蓝色 | 这是重要模块或工程入口 | 子系统入口、API 边界、构建测试入口 |
| `focus` | 关注，黄色 | 容易忽略，最好看一眼 | 生成规则、兼容关系、同步要求、特殊说明 |
| `core` | 核心，红色 | 项目主干，必须优先掌握 | 主入口、核心链路、核心模型、主调度器 |
| `stable` | 稳定，绿色 | 已确认，可以依赖 | 唯一事实来源、权威契约、稳定基准 |
| `extension` | 扩展，紫色 | 新增能力从这里接入 | 注册表、插件、Provider、Adapter、Hook |

### 5.1 核心 `core`

只用于项目骨架：主运行入口、主控制流、核心领域模型和最主要的业务链路。

- 小型项目通常只有 1～3 条。
- 中型项目通常只有 3～6 条。
- 不得超过全部备注的 10%。
- 不能因为文件复杂、经常修改或代码很多就标红。
- 不要把整个 `src` 目录和大量子文件同时标红。

判断问题：如果只能先读几个路径来理解系统，它是否必须在其中？

### 5.2 重要 `important`

用于重要但不属于整个项目主干的模块、边界和工程入口。

例如子系统入口、HTTP API 边界、构建脚本、测试入口、重要架构文档。红色是项目主干，蓝色是重点节点。

### 5.3 关注 `focus`

用于容易被忽略的事实，不表示警告或危险。备注文字必须说明具体需要关注什么。

推荐写法：

```text
需与移动端同步
保留 v1 客户端兼容
由 Schema 生成，勿手改
修改后需更新快照
```

禁止只写“需要注意”“比较复杂”“容易出错”。

### 5.4 稳定 `stable`

用于仓库中可以作为依据的内容：唯一事实来源、权威契约、稳定基准或已确认的设计决策。

绿色表示“可以依赖这个结论”，不表示文件永远不再修改，也不表示修改一定安全。

### 5.5 扩展 `extension`

用于新增同类能力时真正需要接入的位置，例如路由注册、插件注册、Handler 注册、Provider、Adapter、Hook、Strategy 或中间件入口。

必须存在真实扩展机制，不能仅因为目录名包含 `plugins` 或 `extensions` 就标紫色。

## 6. 样式决策顺序

一个路径只能选择一种样式。按以下顺序判断：

1. 它是否属于主运行入口、主执行流或核心领域模型？是则使用 `core`。
2. 它是否主要用于注册或接入新的同类能力？是则使用 `extension`。
3. 它是否是权威来源、唯一事实来源或稳定基准？是则使用 `stable`。
4. 它是否包含开发者容易错过的非显然信息？是则使用 `focus`。
5. 它是否是重要模块、边界、工具或次级入口？是则使用 `important`。
6. 都不是则省略 `style`；没有足够价值时直接省略整条备注。

选择颜色后再次检查：备注文字是否能够解释这个颜色？如果不能，降级为普通或删除备注。

## 7. 常见工程场景

| 工程对象 | 常见样式 | 判断重点 |
| --- | --- | --- |
| 前端或后端主启动文件 | `core` | 是否为主要运行入口 |
| 根路由、根状态容器 | `core` | 是否构成应用主干 |
| 核心领域模型或核心业务服务 | `core` | 是否决定主要业务流程 |
| 子模块入口、Controller、API 边界 | `important` | 是否为重要节点但非全局主干 |
| 构建、测试入口 | `important` | 是否为主要工程入口 |
| README、架构文档 | `important` | 是否值得新人优先阅读 |
| OpenAPI、Proto、Schema | `stable` | 是否为权威契约 |
| 根版本文件、统一配置源 | `stable` | 是否被其他位置统一读取 |
| 生成代码或兼容层 | `focus` | 是否存在容易漏掉的规则 |
| 跨端同步配置 | `focus` | 是否必须同步修改 |
| 路由、插件、Handler 注册表 | `extension` | 是否为实际新增入口 |
| Adapter、Provider、Hook | `extension` | 是否承载可替换或新增实现 |
| 普通工具和测试数据 | 默认 | 是否只有普通职责 |

同一类文件在不同项目中可能使用不同颜色，必须根据仓库证据决定。

## 8. JSON 格式

只允许生成严格 JSON：

```json
{
  "version": 1,
  "notes": {
    ".": {
      "text": "项目整体定位",
      "style": "important"
    },
    "src/main.ts": {
      "text": "服务主入口",
      "style": "core"
    },
    "openapi/api.yaml": {
      "text": "API 权威契约",
      "style": "stable"
    },
    "src/handlers/registry.ts": {
      "text": "新增 Handler 注册",
      "style": "extension"
    }
  }
}
```

格式要求：

- `version` 必须是整数 `1`。
- `notes` 必须是对象。
- 项目根路径使用 `.`。
- 其他 key 使用大小写精确的仓库根相对路径，以 `/` 分隔。
- 只允许备注当前确实存在的文件或目录。
- 禁止绝对路径、反斜杠、重复 `/`、末尾 `/`、`.` 或 `..` 路径段。
- 禁止为 `.codebase-notes.json` 自身添加备注。
- 每条必须有非空 `text`。
- 默认样式应省略 `style`。
- 不添加 `tags`。
- 不添加注释、尾逗号或重复 key。
- 使用 UTF-8、两个空格缩进、LF 换行和一个结尾换行。
- note key 按稳定字典序排列。
- 只有项目已经存在有效 Schema 引用时才保留 `$schema`，不要凭空添加无效地址。

## 9. 更新已有配置

如果 `.codebase-notes.json` 已存在：

- 合并更新，不要直接覆盖用户写的有价值备注。
- 保留不理解的额外字段。
- 修正已经被代码证明确认过时的备注。
- 不确定是否过时的备注继续保留。
- 已移动的路径只有在新位置能够被确认时才更新。
- 删除冗余、重复、纯粹复述文件名或没有证据的 AI 备注。
- 按本文规范重新检查过长文字和颜色分类。

## 10. 完成前校验

1. 使用严格 JSON 解析器验证文件。
2. 检查每个路径真实存在且大小写准确。
3. 检查所有 `text` 非空且不超过 24 个 Unicode 字符。
4. 检查 `style` 只使用本文定义的六个值。
5. 检查 `core` 数量不超过全部备注的 10%。
6. 检查每条彩色备注的文字能够解释其颜色。
7. 检查没有秘密、Token、Cookie、私有地址或个人信息。
8. 检查没有冗余、猜测性或很快会过期的备注。
9. 更新已有文件时检查差异，避免误删用户备注和未知字段。

完成后简短报告：创建还是更新、备注数量、核心路径、稳定基准、扩展点和校验结果。
