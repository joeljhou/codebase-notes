# Codebase Notes 备注图标规范

> 状态：已完成
> 最后更新：2026-07-21
> 适用范围：Codebase Notes v1 备注文本与 AI 初始化
> 关联文档：[AI 初始化规范](S20260720_platform_ai_initialization.md)；[v1 协议](S20260717_platform_codebase_notes_protocol.md)

## 1. 结论

备注统一写成“一个图标 + 一个半角空格 + 一条短语”。图标说明路径在代码地图里的角色，`style` 控制视觉层级；两者是两条信息通道，不互相替代。

```text
<icon> <text>
```

示例：`◆ API 权威契约`、`◎ 服务主入口`、`✚ 新增 Handler 注册`。

## 2. 目标与范围

这套图标用于提高项目树的扫读效率，让人和 AI 能快速区分核心、入口、契约、扩展点、风险和工程动作。它适用于 `.codebase-notes.json` 中的 `notes.*.text`。

本规范不改变 JSON 协议，不新增字段，也不规定 IDE 内的具体颜色。文件类型图标、产品 Logo、状态栏图标和普通文档排版不在范围内。

## 3. 主图标集

只使用下面 12 个主图标。一个图标只表达一种稳定语义，同一条备注只使用一个图标。

| 图标 | 固定语义 | 适合对象 | 示例 |
| ---- | -------- | -------- | ---- |
| `★` | 核心 | 项目主角、主链路、核心领域模型 | `★ 订单核心链路` |
| `◎` | 入口 | 运行入口、阅读入口、接入入口 | `◎ 服务主入口` |
| `◆` | 契约 | Schema、协议、接口定义、跨端约定 | `◆ API 权威契约` |
| `✚` | 扩展 | 插件、Provider、Adapter、Hook、注册点 | `✚ 新增 Handler 注册` |
| `⚠` | 注意 | 风险、兼容问题、易踩坑、非显然限制 | `⚠ 修改需同步双端` |
| `✓` | 稳定 | 已验证规则、共享基准、可靠行为基线 | `✓ 双端行为基准` |
| `▤` | 文档 | 指南、说明、维护文档、文档入口 | `▤ 安装与开发指南` |
| `⚙` | 工程 | 构建、测试、部署、代码生成、配置动作 | `⚙ 全量验证入口` |
| `✧` | 唯一源 | 单一事实来源、版本源、生成源头 | `✧ 版本唯一源` |
| `🔒` | 边界 | 安全、权限、模块边界、隔离约束 | `🔒 认证边界` |
| `🚀` | 发布 | 发布、交付、制品、上线流程 | `🚀 双端发布包汇总` |
| `⊗` | 废弃 | 已废弃实现、禁止依赖、迁移前旧入口 | `⊗ 已废弃适配层` |

## 4. 选择规则

按路径最值得被记住的角色选择图标，而不是按文件扩展名机械选择：

1. 主执行链路或核心领域对象用 `★`。
2. 进入系统、模块或阅读路径的起点用 `◎`。
3. 定义跨模块约定的内容用 `◆`；若重点是“唯一权威来源”，改用 `✧`。
4. 确实能添加同类能力的位置用 `✚`，普通目录不能因为未来可能扩展就使用它。
5. 已验证的测试基准或稳定规则用 `✓`；不要用它表示任务完成。
6. 非显然风险或同步要求用 `⚠`；不要用它泛指“重要”。
7. 构建、测试、部署等执行入口用 `⚙`，正式交付物或发布流程用 `🚀`。
8. 其余角色按主图标集的固定语义选择；没有足够明确的角色时不添加该备注。

路径同时符合多个角色时，只保留对导航最有用、最不显然的一个。例如权威 Schema 优先写 `✧ 配置格式唯一源`，不堆叠成 `◆✧`。

## 5. 与 `style` 的关系

图标回答“它是什么角色”，`style` 回答“浏览项目树时应给它多少视觉权重”。不要根据图标自动推导 `style`，也不要用颜色替代图标。

| 例子 | 图标含义 | `style` 含义 |
| ---- | -------- | ------------- |
| `★ 订单核心链路` + `core` | 主链路 | 项目骨架 |
| `✚ 新增 Provider 注册` + `extension` | 扩展点 | 优先发现的扩展位置 |
| `✧ API 唯一契约` + `stable` | 唯一源 | 可靠基线 |
| `⚠ 修改需同步客户端` + `focus` | 风险 | 容易漏掉的事实 |
| `⚙ 全量验证入口` + `important` | 工程动作 | 重要工程入口 |

同一个图标可以搭配不同样式。例如 `◎ README 阅读入口` 可以不设样式，而 `◎ 服务主入口` 可以使用 `core`。

## 6. 文本格式

- 每条备注必须以主图标集中的一个图标开头，后接一个半角空格。
- 图标后只写一个事实，默认使用简体中文，代码标识符保持原样。
- 整条 `text` 建议为 4–16 个字符，最多 24 个 Unicode code point；长度包含图标和空格。
- 不使用句号、第二个图标、星级、颜文字或额外强调符号。
- 不以“该文件”“该目录”“这个文件”或“主要负责”开头。
- 不复述文件名中已经显而易见的信息。

正确：

```text
★ 支付核心链路
◆ Webhook 数据契约
⚠ 修改后需重新生成
```

错误：

```text
⭐🔥 超级重要入口
⚠重要配置
✓ 已完成
📂 src 目录
```

## 7. 更新与兼容

- 新增或由 AI 重写的备注必须符合本规范。
- 更新已有配置时，将没有规范图标的旧备注补齐；语义不明确时先从代码证据确认，不靠路径名猜测。
- 保留用户写的有效事实和未知 JSON 字段，只规范化 `text` 的图标前缀。
- 旧备注中的非主图标应替换为语义最接近的主图标；不能可靠映射时重新判断备注角色，仍无法判断则删除低价值备注。
- 图标是普通 Unicode 文本。读取端不得依赖字形宽度、颜色或 emoji 渲染方式解释语义。

## 8. 验收清单

- [ ] 每条备注只有一个主图标，且图标后是一个半角空格。
- [ ] 图标符合路径的真实角色，并有代码或文档证据。
- [ ] `⚠` 只用于风险，`✚` 只用于真实扩展点，`✓` 不表示完成态。
- [ ] 图标和 `style` 分别表达角色与视觉层级。
- [ ] 整条文本不超过 24 个 Unicode code point。
- [ ] 没有重复、猜测性或很快过期的备注。
